# v0.3 方案：Agent 调用式转发

日期：2026-07-07

## 策划层评审补充（ZCode 视角，2026-07-08）

本节由 ZCode（策划角色）在 Codex 设计稿基础上补充产品/风险判断，供负责人拍板。

### 实施顺序建议：先 M3.2 文本标记，后 M3.1 MCP

Codex 原稿写「MCP 优先、文本兜底」。从交付风险看，建议反过来：

| 维度 | M3.2 文本标记 | M3.1 MCP |
|---|---|---|
| 改动范围 | broker 单端（monitor 解析 + 复用 dispatchBridge） | 新建 sidecar 进程 + 本地控制通道 + token + workspaceFingerprint |
| 复用现有能力 | 几乎全部复用 | 需新建 sidecar，发送仍回扩展（架构桥接） |
| 三端一致 | 一套协议通吃 | 每端都要验证 MCP 挂载（ZCode 已验，Codex/Claude 未验） |
| 见效速度 | 1-2 天 | 1 周+ |
| 风险面 | 小（纯解析） | 大（本地端口/token/进程间通信全链路） |

理由：M3.2 让「agent 主动调用转发」这个核心价值最快可用，且对所有 agent 通用、不依赖各端 MCP 挂载成熟度。M3.1 作为体验升级（agent 原生工具调用，比输出标记块更自然），在 M3.2 稳定后再上，避免一开始就背 sidecar 架构的全套风险。

**结论建议：M3.2 先行，M3.1 跟进。** 若负责人更看重「原生工具调用体验」且能接受更长周期，也可维持 Codex 原序。

### plan mode 拦截：产品决策点

Codex 实测发现：ZCode plan mode 下，未声明 `readOnlyHint` 的工具被拦截。`broker_forward` 有副作用，不能伪装只读。

两种产品取舍：

- **A. 接受拦截**：plan mode 本就是「只规划不执行」，转发属于执行动作，被拦合理。用户切到执行模式再转发。文案提示：「当前处于 plan mode，转发需切换到执行模式」。**低改动，符合 plan 语义。推荐。**
- **B. 工作区白名单**：给 `broker_forward` 配置「受信任工具」，plan mode 放行。破坏 plan mode 安全语义，不推荐。

建议 A。文档已按 A 写。

### 身份前缀要加能力说明（影响现有配置）

M3.2 需要在 agent 身份前缀里加一句「如何发起转发」。这会动到用户已配置的 red/blue 前缀文本。两个处理：

- 默认不自动改用户前缀，提供一段「建议附加语」让用户自行粘贴（尊重用户既有定制）。
- 或在 Broker 层拼接（前缀 + 能力说明分开存），不动用户文本。

建议后者：能力说明作为系统级常量拼接，不污染用户身份锁文本。

## 纠偏结论

v0.3 不是 auto-pong，也不是让两个 Agent 全自动互聊。

正确目标是：用户在某个 Agent 中提出“发送给另一个 Agent”的要求时，该 Agent 可以调用 Broker 提供的转发能力，由 Broker 完成一次明确的跨端发送。

典型场景：

```text
用户在 ZCode 中说：把这个方案发送给 Codex 评审。
ZCode 生成一次 Broker 转发调用。
Broker 将指定内容发送到 Codex。
Codex 正常回复。
流程到此结束，不自动把 Codex 回复再发回 ZCode。
```

## 产品边界

做：

- 一次 Agent 调用触发一次 Broker 转发。
- 支持目标：Codex、ClaudeCode、ZCode。
- 支持来源：当前已监控的 Codex、ClaudeCode、ZCode 会话。
- 支持人工可见状态：待调用、已转发、失败、被拦截。
- 转发时复用现有三端发送通道。
- 防止重复执行同一条调用。

不做：

- 不做自动来回接力。
- 不做“双边 DONE”循环。
- 不做模型互相追问到终止。
- 不扩大到多跳路由。
- 不在没有用户授权的情况下让 Agent 任意控制所有目标。

## MCP 实测结论

2026-07-07 已在本机 ZCode 3.2.5 做最小 MCP 探针验证：

- ZCode 会读取工作区级 `.zcode/config.json` 的 `mcp.servers`。
- ZCode 会以 stdio 子进程启动 MCP server，并发送 `initialize`、`notifications/initialized`、`tools/list`。
- ZCode agent 能在模型回合中真实触发 `tools/call`。
- 工具调用返回值会进入 ZCode assistant 回复。
- 当前 ZCode 处于 plan mode 时，未声明只读的 MCP 工具会被拦截；工具必须提供 `annotations.readOnlyHint=true` 才能作为只读工具放行。

实测 marker：

```text
BROKER_MCP_PROBE_20260707_2335_READONLY
```

探针收到：

```text
method: tools/call
name: broker_probe_ping
text: BROKER_MCP_PROBE_20260707_2335_READONLY
```

ZCode 最终回复：

```text
BROKER_MCP_PROBE_OK:BROKER_MCP_PROBE_20260707_2335_READONLY
```

因此 v0.3 的主路径可以从“文本标记优先”调整为“MCP 优先，文本标记兜底”。

### 2026-07-08 复测：9224 启动方式不是 MCP 失败根因

ZCode 复测提出假设：Broker 需要的 `--remote-debugging-port=9224` 调试实例可能走了精简初始化，导致 MCP 工具只进入 capability registry，没有注入 agent 模型工具集。

Codex 在当前 9224 调试实例中做了最小对照：

- 当前 ZCode 主进程：`D:\APP\ZCode\ZCode.exe --remote-debugging-port=9224`。
- 临时在 `C:\Users\pengq\.zcode\cli\config.json` 增加 read-only MCP server `brokerProbe9224`，测试后恢复原配置。
- 在 9224 实例中为 `d:\python\broker-chat-vscode` 新建干净会话 `sess_95d87d24-15c0-4e67-8e98-1bac1250624a`。
- 通过 CDP 发送用户请求，要求调用 `mcp__brokerProbe9224__broker_probe_ping`。
- 独立 MCP server 日志收到真实调用：

```text
event=tool_call
name=broker_probe_ping
text=BROKER_MCP_9224_NEWSESSION_1783484713599
```

结论：

- `--remote-debugging-port=9224` 本身不会阻止 ZCode agent 自主调用 MCP 工具。
- 同一 9224 实例的新建会话可以真实 `tools/call`。
- 旧会话 `sess_42d549a0-2630-49f0-b3d0-55d6c8d5473a` 的 rollout 请求体仍只有 15 个内置工具（`Agent/Bash/Read/.../SendMessage/ReadSessionContext`），没有任何 `mcp__*`；在该旧会话里复测不触发 MCP。
- 因此当前更可靠的判断是：**MCP 工具注入与会话/回合初始化状态有关**。已有旧会话不一定会获得后来新增的 MCP 工具；新建会话在同一 9224 实例下可以获得并调用。

产品影响：

- MCP 方案可保留，但不能承诺“任意旧会话立即可调用新 MCP 工具”。
- Broker UI/文档需要提示：配置 MCP 后，ZCode 侧可能需要新建/重开目标对话才能让 agent 看见 `broker_forward` 工具。
- 若必须支持旧会话无缝转发，仍需要 M3.2 文本标记兜底。

  `D:\python\broker-chat-vscode\.broker-chat\runtime.json`。
- ZCode 全局 MCP 配置 `C:\Users\pengq\.zcode\cli\config.json` 已注册 `brokerForwardZCode`：

  "command": "node",

    "--workspace",
    "D:\\python\\broker-chat-vscode"

真实调用链：

```text

  -> mcp__brokerForwardZCode__broker_forward
  -> dist/mcp/brokerMcpServer.js
  -> HTTP POST 127.0.0.1:<runtime.port>/agent-command
  -> BrokerControlServer
  -> BrokerController.forwardAgentInvokedCommand
  -> OfficialUiBridge foreground guard
  -> Codex 官方面板
```

成功 marker：

```text
requestId: probe.mcp.real.000005
tool result: BROKER_FORWARD_OK:probe.mcp.real.000005:已桥接发送到 Codex 官方面板。
Codex 收到文本: [full-chain probe via real MCP tool, 请忽略, 无需回复]
```

结论：

- ZCode agent 可以真实调用 `broker_forward` MCP 工具。
- sidecar 可以读取 runtime 文件并调用 VS Code 扩展本地控制端点。
- VS Code 扩展可以通过既有官方面板发送链路把内容送到 Codex。
- 这不是手动复制、不是文本标记兜底、也不是 app-server 写历史；它走的是 v0.3 MCP 生产路径。

前两次失败根因：

- 当用户还在 ZCode 窗口里和 ZCode agent 交互时，Windows 前台窗口是 `ZCode.exe`。
- `OfficialUiBridge` 在发送到 Codex / ClaudeCode 前会校验 VS Code 是否为当前前台工作区窗口。
- 前台不是 VS Code 时守卫拒绝发送，这是预期安全行为，不是 MCP 注入失败。
- 成功那次的必要条件是：MCP 调用到达 Broker 时，VS Code 窗口正好处于前台，守卫验证通过。

产品约束：

- `ZCode -> Codex/ClaudeCode` 仍受 VS Code 官方面板发送机制限制，要求 VS Code 工作区窗口处于前台。
- `Codex/ClaudeCode -> ZCode` 走 ZCode CDP 注入，不受 VS Code 前台守卫限制。
- 后续产品化应把前台限制做成可恢复状态，例如返回 `BROKER_FORWARD_NEEDS_FOREGROUND` 并保留待发送请求，用户切回 VS Code 后再继续发送。

不要重复测试的误区：

- 不要用旧 ZCode 会话是否看得见 MCP 工具来否定 MCP；新增工具可能只注入新会话。
- 不要把 plan mode 拦截有副作用工具解释成 MCP 不可用；`broker_forward` 本来就不应声明只读。
- 不要把前台窗口不是 VS Code 导致的 guard 拒绝解释成 sidecar 或本地控制通道失败。
- 不要改松焦点守卫来追求“后台无感发送”；这会增加发错窗口、发错会话的风险。

## 两阶段实现策略

### M3.1 MCP Tool Bridge

优先做真正的 MCP 工具入口：

- Broker 提供本地 MCP server sidecar。
- 工具名：`broker_forward`。
- 参数：
  - `target`: `codex | claude | zcode`
  - `content`: string
  - `mode`: `message`
  - `requestId`: string
- 返回：
  - `accepted`
  - `target`
  - `deliveryState`
  - `error`

MCP 只作为标准调用入口，不替代 Broker 的三端发送能力。

推荐架构：

```text
ZCode / Claude / Codex
  -> stdio MCP sidecar: broker-mcp-server
  -> localhost control endpoint with auth token
  -> VS Code Broker extension
  -> existing dispatchBridge()
  -> Codex / Claude / ZCode
```

原因：

- MCP server 通常是独立 stdio 进程，不应直接依赖 VS Code extension host。
- VS Code Broker extension 已经拥有三端发送能力：Codex / ClaudeCode 走官方面板命令 + SendKeys，ZCode 走 CDP。
- 外部 sidecar 不应重新实现所有发送细节，否则会复制最脆弱的焦点和 CDP 逻辑。
- 现有监控时间线仍由 Broker extension 负责，MCP 调用只是多一个触发来源。

sidecar 启动参数建议：

```text
broker-mcp-server --source zcode --broker-workspace D:\python\broker-chat-vscode
broker-mcp-server --source codex --broker-workspace D:\python\broker-chat-vscode
broker-mcp-server --source claude --broker-workspace D:\python\broker-chat-vscode
```

source 不由模型参数填写，必须来自 MCP server 配置或启动参数，防止 agent 冒充其他来源。
`sourceSessionId` 由 Broker extension 结合当前监控快照推断；如果当前 source 没有可用会话，则拒绝转发并返回真实错误。

本地控制通道：

- 扩展启动时创建 `127.0.0.1` 本地 HTTP server 或 Windows named pipe。
- 扩展把 `port/pipe + token + workspaceFingerprint` 写入一个只给本机读取的 runtime 文件。
- sidecar 启动时按 `--broker-workspace` 找 runtime 文件，并携带 token 调用扩展。
- 请求必须绑定 workspaceFingerprint，避免多个 VS Code 窗口互相串项目。
- token 只用于本机 sidecar 到扩展的控制通道，不写入日志和 UI。

扩展侧新增受控入口，而不是把 `dispatchBridge()` 直接暴露给外部：

```text
BrokerController.forwardAgentInvokedCommand({
  sourceAgent,
  sourceSessionId,
  target,
  content,
  requestId
})
```

职责：

- 验证本地控制请求 token。
- 校验 target 在当前 red/blue pair 中。
- 拒绝 target 等于 sourceAgent。
- 校验 content 非空、长度上限、requestId 格式。
- 按 `sourceAgent + sourceSessionId + requestId` 去重。
- 将状态写入 bridge state / timeline diagnostics。
- 通过内部 `dispatchBridge(target, prompt, "agent-command")` 执行发送。

工具安全语义：

- `broker_forward` 有副作用，不应声明 `readOnlyHint=true`。
- plan mode 下如果 ZCode 拦截有副作用工具，这是合理行为；用户需要在可执行模式下要求 agent 转发。
- 工具描述必须明确：只在用户明确要求转发时调用，一次调用只转发一次，不自动循环。
- 每次调用必须携带 `requestId`，Broker 按 `sourceAgent + sourceSessionId + requestId` 去重。

### M3.2 Transcript Command Bridge Fallback

保留不依赖 MCP 的文本调用协议作为兜底。原因：

- Codex / ClaudeCode 侧的 MCP 挂载体验仍需分别验证。
- 如果某端处于 plan/approval 模式或用户未配置 MCP，文本标记仍可提供显式、可审计的一次性转发。
- 现有 Broker 已经能读取三端真实会话并向三端发送。

Broker 监听已监控会话中的新消息，识别显式工具调用块：

```text
BROKER_FORWARD_BEGIN
target: codex
mode: message
id: <agent-generated-id>

<要发送的正文>
BROKER_FORWARD_END
```

规则：

- `target` 只能是 `codex`、`claude`、`zcode`。
- `target` 不能等于来源 Agent。
- `id` 用于去重，同一来源会话内只执行一次。
- 正文为空则拒绝。
- 正文长度超过上限则拒绝。
- 调用块必须来自 baseline 之后的新消息。
- 默认只接受当前桥接对里的两个 Agent 互相调用；第三端只显示“目标未在当前配对中”。
- 执行失败要写入状态，不无限重试。

这需要给 Agent 的身份前缀加一句能力说明。例如给 ZCode：

```text
当用户明确要求你把内容发送给 Codex / ClaudeCode 时，不要口头声称已发送。
请输出 BROKER_FORWARD_BEGIN / BROKER_FORWARD_END 调用块，由 Broker 实际转发。
```

## 与现有 AutoForwardEngine 的关系

不要把这个功能塞进 `AutoForwardEngine`。

原因：

- AutoForwardEngine 是关键词自动转发，语义是“检测到用户关键词后等模型回复再转发回复”。
- Agent 调用式转发的语义是“检测到一次工具调用后，发送调用正文”。
- 两者的 baseline、去重、失败语义不同。

文本标记兜底模块建议：

```text
src/automation/AgentCommandBridgeEngine.ts
```

职责：

- 扫描新消息中的 `BROKER_FORWARD_*` 调用块。
- 校验目标、来源、长度、配对、去重。
- 输出 `AgentCommandDecision`。
- 由 `BrokerController` 复用 `dispatchBridge()` 执行发送。

## 状态机

```text
idle
  -> detected
  -> sending
  -> sent
  -> failed
  -> blocked
```

阻断条件：

- 目标不在当前配对中。
- 目标等于来源。
- 正文为空。
- 正文过长。
- 同一 `sourceAgent + sessionId + requestId` 已执行过。
- 当前 bridge 正在忙。
- 目标发送通道未就绪。

## 测试要求

必须先写测试：

- MCP sidecar 能返回 `tools/list` 中的 `broker_forward` schema。
- MCP sidecar 收到 `tools/call` 后能向扩展控制端点发送一次受控请求。
- 控制端点缺 token、错 token、workspaceFingerprint 不匹配时拒绝。
- `source` 只能来自 sidecar 启动参数，模型参数不能覆盖 source。
- `requestId` 重复时只执行一次。
- MCP 工具返回真实失败原因，不吞错为成功。
- 能解析合法调用块。
- 文本标记同一 requestId 只执行一次。
- target 等于 source 时拒绝。
- target 不在当前 pair 时拒绝。
- 空正文拒绝。
- 超长正文拒绝。
- baseline 前历史调用不执行。
- 发送失败后不会无限重试。
- Agent command 运行时不触发 auto-forward 自激。

## v0.3 验收

- 用户在 ZCode 中要求“发送给 Codex”。
- ZCode 调用 `broker_forward` MCP 工具。
- Broker sidecar 把调用转交给 VS Code 扩展。
- Broker 扩展校验后把正文发送到 Codex。
- Codex 收到真实消息并开始回复。
- Broker 不自动把 Codex 回复发回 ZCode。
- 状态栏显示一次调用的结果。
